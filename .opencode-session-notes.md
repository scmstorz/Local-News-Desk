## Gefixte Punkte

### 1. Embedding-Fehler bei Unicode-Titeln

Problem:

```text
Embedding generation failed for article 18701: Ollama embedding response did not contain a usable vector
```

Ursache:

- Artikel `18701` hatte kyrillischen/Unicode-Titel.
- `build_embedding_input_text()` nutzte vorher die ASCII-Titelnormalisierung.
- Ergebnis war leerer String `''`.
- Ollama bekam leeren Input und lieferte keinen nutzbaren Vector.

Fix:

- `build_embedding_input_text()` hat jetzt Unicode-freundlichen Fallback.
- Leere Embedding-Inputs werden in `select_article_for_embedding()` uebersprungen.
- Ollama-Embedding-Response-Parser akzeptiert jetzt:
  - `embedding: [...]`
  - `embeddings: [[...]]`
  - `embeddings: [...]`
  - `data[0].embedding`

Verifikation:

- Artikel `18701` erzeugte danach einen echten Input:

```text
'Сэм Альтман: «Хатоларимиз учун аҳолидан чуқур узр сўрайман»'
```

- In der DB wurde fuer `18701` ein Embedding gespeichert:

```text
18701 | nomic-embed-text-v2-moe:latest | length 10277 | 2026-04-26T10:50:10Z
```

### 2. Summary nach `s` im Feed

Problem:

- User drueckte `s` im empfohlenen Feed und hatte den Eindruck, dass keine Summaries entstehen.
- DB zeigte: Artikel `18556` wurde requested, processing, dann ready.
- Also war API grundsaetzlich aktiv, aber Summary-Arbeit konnte durch Embedding-Worker/Ollama-Lock verzoegert wirken.

Fix:

- `RuntimeState` hat jetzt `summary_event`.
- `set_feed_decision(..., "summarize")` ruft nach DB-Update `wake_summary_worker()`.
- `summary_worker()` wartet auf Event statt nur Polling.
- `embedding_worker()` pausiert, sobald Summary-Arbeit `queued` oder `processing` ist.
- `/api/articles/<id>/summarize` behandelt `RuntimeError` jetzt sauber als HTTP 409.

Verifikation:

- Artikel `18556` ist ready/unreviewed:

```text
SpaceX und OpenAI planen IPOs – Historie mahnt zur Kursvorsicht
```

### 3. `Empfohlen` und `Vielleicht` zeigten doppelte Artikel

Problem:

- User sah wahrscheinlich dieselben Artikel in `Empfohlen` und danach in `Vielleicht`.
- Ursache war UI-Definition: Modus hiess `Vielleicht+` und zeigte bewusst `Empfohlen + Vielleicht`.

Fix:

- Frontend-Button heisst jetzt `Vielleicht`.
- Frontend filtert `Vielleicht` nur noch mit `prediction.tier === "maybe"`.
- Backend unterstuetzt `mode=maybe`.
- `maybe_plus` bleibt backendseitig als Kompatibilitaetsmodus vorhanden.

Verifikation mit aktuellem Feed:

```text
recommended 14
maybe 35
overlap 0
```

Zusaetzlich wurde der sichtbare Feed auf exakte/nahe Titelduplikate geprueft:

```text
exact-normalized-dupes 0
near pairs 0
```

### 4. `S`-Taste im Feed kam nicht mehr zuverlaessig an

Problem:

- User meldete, dass mehrfaches Druecken von `S` im `Empfohlen`-Feed nichts ausloest.
- DB bestaetigte: Nach `2026-04-26T10:59:00Z` gab es weitere `article_skipped`-Events, aber keine neuen `summary_requested`-Events.
- Damit war klar: Backend/Worker waren nicht der primaere Fehler; der Tastendruck kam im Frontend nicht robust im `handleFeedSummarize()`-Pfad an.

Fix:

- Keyboard-Listener laeuft jetzt in der Capture-Phase:

```js
document.addEventListener('keydown', handleKeyboard, true);
```

- Shortcut-Erkennung akzeptiert nicht nur `event.key === "s"`, sondern auch die physische Taste `event.code === "KeyS"`.
- Behandelte Shortcut-Events rufen `preventDefault()` und `stopPropagation()` auf.
- Editable Targets (`input`, `textarea`, `select`, `contenteditable`) werden ausgenommen.
- `event.repeat` wird ignoriert, damit gehaltene Tasten nicht mehrere Artikel versehentlich wegschieben.
- Beim Druecken von `S` passiert sofort sichtbar etwas:
  - Toast `Summary vorgemerkt.`
  - Feed-Karte wird sofort entfernt bzw. Empty-State gerendert
  - sichtbarer Status `Summary vorgemerkt: ...`
- Wenn kein aktiver Feed-Artikel vorhanden ist, gibt es jetzt einen Toast statt stillem Return.

Wichtig:

- User muss `local-news-app.html` im Browser hart neu laden, damit der neue Shortcut-Code aktiv ist.
